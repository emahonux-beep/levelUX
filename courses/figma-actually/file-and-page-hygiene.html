File & Page Hygiene

schedule 6 min read
🎬 On-screen: title card, "File & Page Hygiene"

This lesson isn't glamorous, but it's the one that saves you the most time down the road. A messy file doesn't just look bad — it slows down everyone who has to work in it after you, including future-you in six months.

Here's a page structure that holds up on real projects.

A page structure that scales

🎬 On-screen: side panel of a Figma file, scrolling through the page list as each one is named
  1. Cover page. A presentation-style frame with the product or feature name, who's responsible, and its status. Set this as the file thumbnail so you can identify the file at a glance instead of squinting at tiny file names in the dashboard.
  2. Divider. A page with a few dashes in the name to create a visual separator line in the page list. Small trick, real payoff.
  3. Noodles / Explorations. Your scratch pad. This page can — and should — get messy. Throw ideas at the wall here.
  4. Responsive design. Kept separate from native, since different teams often own each, and there are real platform-specific best practices that don't cross over cleanly.
  5. Native design. Same logic, opposite platform.
  6. Divider.
  7. Supporting pages. Local components, demos, or research artifacts — whichever apply to the project.
  8. Archive / Do Not Use. The last page in every file. Nothing gets deleted here; it just moves out of the way.

Explorations don't have to disappear to keep a file clean. They just need a place to live that isn't the main path.

Why the archive page instead of just deleting things

You never know when you'll want to revisit an idea that got shelved. Six to twelve months out, "the direction we didn't take" is often exactly what someone asks about. Pair the archive page with branches — branch and merge management keeps a real version history of explorations without cluttering your main pages. Nothing to dig through, nothing lost either.

Messy where it should be messy, clean where it counts

Your Noodles page can be chaos. Layers everywhere, half-finished ideas, things that don't line up. That's the point — it's where you're allowed to think out loud.

The responsive and native pages are a different story. That's where organization actually matters:

  • Use sections to group related frames instead of letting everything float loose.
  • Name your frames and layers. "Frame 47" tells the next person nothing. "Checkout — Confirmation" tells them everything.
  • One frame per full page, generally. Instead of duplicating an entire page to show variant states, build sections beneath the main frame for individual components that change.

The step 1 / 2 / 3 problem

🎬 On-screen: example — a three-step onboarding flow, shown as one frame with three component variants beneath it instead of three duplicated pages

Say you've got a flow with steps one, two, and three, and the content changes at each step. Instead of recreating the whole page three times, make a copy of just the component that changes — the content for each step lives in that one component, not in three duplicated pages.

This keeps file size down and keeps you from copy-pasting the same page over and over just to show a small content change. Newer prototyping tools are starting to reduce how often you need to do this by hand, but for now, when you are hands-on in the file, this habit pays for itself quickly.
