Module 4 · File Management · Lesson 9 of 10

Branching

schedule 4 min read
🎬 On-screen: title card, "Branching"

We referenced branching back in Lesson 3 as the partner to your archive page. This lesson goes deeper — what a branch actually is, when to reach for one, and how to bring it back into your main file.

What a branch is

A branch is a separate copy of a file where you can explore, experiment, or make changes without touching the main version. Nothing you do on a branch affects the file everyone else is working from until you decide to merge it back.

🎬 On-screen: create a branch from an existing file, show the branch indicator in the file name

When to use one

  • Bigger explorations you don't want cluttering your Noodles page or risking the stability of the main file.
  • Parallel work — when you and someone else need to work on meaningfully different directions at the same time without stepping on each other.
  • Anything experimental that might not ship, but that you don't want to lose either. This is exactly the safety net for those "we might want this again in six to twelve months" ideas.

A branch means you can explore fearlessly without putting the main file at risk. That freedom is the whole point.

Merging a branch back

When a branch is ready — the direction is validated, the exploration is done, the work is approved — you merge it back into the main file. Figma will show you what's changed, which gives you a chance to review before the merge finalizes rather than merging blind.

🎬 On-screen: walk through a merge review screen, pointing out what changed

If a branch doesn't end up going anywhere, that's fine too. It doesn't need to be merged just because it was created — an unmerged branch is still a preserved record of the idea, in case it's worth revisiting later.

What's next
